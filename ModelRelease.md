# Model Release - Reusing Local Capabilities
This repo is intended to automatically build the models that will be used for 
transformation into FHIR. However, if there is interest in accessing the 
data from DBT using SQL Alchemy, you could incorporate this into the repo to 
generate the models as part of the release. 

By making the changes listed below, other models will:
* Build Harmony files from enumeration's "meaning"
  * Add selected subset of those columns as a page in the docs
  * Allow the user to down the full CSV from that page
  * Add those files to final zip file
* Build the SQL Alchemy models and "publish" them as part of a release on
  * Creation of a new tag (must start with v)
  * Publishing a new release from within GH website
  * Dispatch action for manual publication.
* Build the data-dictionary files. These should be available inside zip files once the gh action is completed

## GitHub Action - release.yaml
In addition to zipping up your project directory, it will also build the 
package up the SQL Alchemy files and add them to the release downloads. 

Basically, just drop [Release Action](https://github.com/carrollaboratory/kfi-fhir-input/blob/main/.github/workflows/release.yaml) into your .github/workflows directory. 

If you don't care about the SQL Alchemy files, you probably can get away with
just removing the two lines below from the workflow file to get an otherwise 
functional release: 

```
      dist/*.whl
      dist/*.tar.gz
```

## Scripts
The scripts directory provides two python script scripts and a github action config: 
| File | Destination | Purpose |
|---|---|---|
| [hatch_build.py](https://github.com/carrollaboratory/kfi-fhir-input/blob/main/scripts/hatch_build.py) | `scripts/hatch_build.py` | Preps the SQL Alchemy file for packaging by twine/build,  which linkml provides tooling. For this script to work, you must configure the toml and just files as described below.  |
| [gen_harmony.py](https://github.com/carrollaboratory/kfi-fhir-input/blob/main/scripts/gen_harmony.py) | `scripts/gen_harmony.py` | enerates a MD style harmony file mapping enumerated values in the model to their expected forms for use in FHIR (such as OMB Race, etc). This assumes that your incoming data complies with those enumerated values. |
| [release.yaml](https://github.com/carrollaboratory/kfi-fhir-input/blob/main/.github/workflows/release.yaml) | `.github/workflows/release.yaml` | Builds Alchemy models for publication within the release alongside the full projection articifact list as a zip file. |


## Changes Required in `pyproject.toml`
### Add build dependencies
My build assumes we are using hatchling for the build. If your template used
something other than hatchling, then this may not work at all. I believe the
only addition I made was adding "tomli" to the end, which is necessary for the
python script to dynamically determine the project's name (to make this more
portable across models.)

```toml
[build-system]
requires = ["hatchling", "uv-dynamic-versioning", "tomli"]
build-backend = "hatchling.build"
```

### dependency-groups
Add the last two lines to your dev value in [dependency-groups]:

```toml
[dependency-groups]
dev = [
    "linkml>=1.9.3",
    "mkdocs-material>=8.2.8",
    "mkdocs-mermaid2-plugin>=1.1.1",
    "jupyter>=1.0.0",
    "mknotebooks>= 0.8.0",
    "ftd-dd @ git+https://github.com/carrollaboratory/ftd-dd",
    "hatchling>=1.27.0",
    "tomli>=2.0.0"
]
```
Please note that third from the last line is what we use to build data 
dictionaries that are used for our DBT pipeline. So, you may or may not need
that line. The others were populated by the template copy. 

### Register the build hook
To tie the script into the 'uv build' system, you have to add the following
lines

```toml
# Release the SQL Alchemy files
[tool.hatch.build.hooks.custom]
path = "scripts/hatch_build.py"
```

### Include required build artifacts and exclude potential problems/warnings
The include list below is required for things to work properly. The exclusions 
are helpful to reduce uv warnings. 
```toml
[tool.hatch.build.targets.sdist]
exclude = [
  ".direnv",
  ".venv",
  "scratch"
]
include = [
  "project/sqlalchemy/",
  "scripts/",
]
```

## Changes Required in `.gitignore`

Add the generated package directory to `.gitignore` — this is created by
`hatch_build.py` as a staging area during builds and should not be committed:

```
# Generated by hatch_build.py during wheel builds - do not commit
<your_schema_name>/
```

Where `<your_schema_name>` matches the `name` field in `pyproject.toml`
(e.g. `kfi_fhir_input/` or `acr_harmonized_data_model/`).

## project.justfile
I would recommend just grabbing a copy from [this repo](https://github.com/carrollaboratory/kfi-fhir-input/blob/main/project.justfile) and editing as needed. 

## justfile
I recommend adding our dependencies project specific stuff to gen-project, 
since that is a depdency for site. 

```
site: gen-project lint gen-doc _gen_ftddd _gen_sqla _gen_harmony
```

Use whichever makes sense for your needs.

### always use 'uv run'
In some earlier versions of my just files (including project.just) I was 
writing to work on my latop. For any dependencies to be found, any scripts
must be run via uv run, and not bare python. This ensures that any dependencies
will already have been installed before any 'just' targets are called.

## How the Build Works

```
uv sync --group dev --no-install-project   # installs linkml/gen-sqla into .venv
                                            # without triggering the editable install
uv sync --group dev                         # now installs the project itself,
                                            # hatch_build.py fires and finds gen-sqla
just site                                   # generates all artifacts including
                                            # project/sqlalchemy/ and project/harmony/
uv build                                    # hatch_build.py sees sqla file already
                                            # exists, skips generation, just bundles
```

The two-step `uv sync` is necessary because hatchling's editable install
fires `hatch_build.py` before `gen-sqla` would otherwise be available.
The first sync installs dependencies only; the second installs the project
once `gen-sqla` is on PATH.

## Verifying the Wheel Contents

After `uv build`, confirm the generated files made it into the wheel:

```bash
unzip -l dist/*.whl | grep -E "\.py|\.csv"
```

You should see entries like:
```
<schema_name>/<schema_name>.py
<schema_name>/harmony.csv        # only if harmony CSVs exist
```

## Installing the Wheel in a Downstream Project

Once a release is published, downstream projects install it directly from
the GitHub Release URL — no package registry required:

**With pip:**
```bash
pip install https://github.com/ORG/REPO/releases/download/v1.0.0/schema_name-1.0.0-py3-none-any.whl
```

**In `requirements.txt`:**
```
schema_name @ https://github.com/ORG/REPO/releases/download/v1.0.0/schema_name-1.0.0-py3-none-any.whl
```

**In `pyproject.toml`:**
```toml
dependencies = [
  "schema_name @ https://github.com/ORG/REPO/releases/download/v1.0.0/schema_name-1.0.0-py3-none-any.whl",
]
```

**Accessing the SQLAlchemy model and harmony CSV in Python:**
```python
from importlib.resources import files

# Import the SQLAlchemy module
from schema_name import schema_name

# Access the harmony CSV as a file (if present)
harmony_csv = files("schema_name").joinpath("harmony.csv")
with harmony_csv.open("r") as f:
    content = f.read()
```
